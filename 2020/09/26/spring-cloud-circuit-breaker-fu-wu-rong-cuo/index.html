Spring Cloud Circuit Breaker 服务容错

Java Spring Cloud Resilience4j
发布日期:   2020-09-26
文章字数:   2.9k
阅读时长:   11 分

参考 Spring Cloud 官方文档的描述，Spring Cloud Circuit Breaker 提供了不同断路器实现的抽象，它为开发者选用不同的断路器实现提供了标准化的 API，在实际的项目应用场景中，我们可以基于 Spring Cloud Circuit Breaker 采取低耦合的方式将第三方断路器集成到项目中，而不直接强依赖于某一种实现，我们知道，通过制定标准，而不强制实现，是 Spring 一以贯之的核心编程思想。目前官方提供四种标准实现，分别是 Netfix Hystrix，Resilience4J，Sentinel，以及 Spring Retry，其中 Netfix Hystrix 属于 Spring Cloud 最早支持的一种容错方案，目前已经进入维护模式，此处不再推荐使用，目前官方优先推荐的是 Resilience4J 方案，它借鉴了 Hystrix 的设计思想，同时充分应用了 Java8 函数式编程的 API，由于它只有很少的外部依赖，是一个非常轻量化的容错框架，而且其中很多函数式思想的运用非常值得我们去感悟。另外，Sentinel 作为阿里微服务开源生态的一部分，Spring Cloud Alibaba 对其做了较为完善的适配，在某些功能性上相比于其他方案甚至更加健壮，在方案选型时也是非常值得我们去考虑的。

Spring Cloud 层面的标准化

由于各种容错框架的实现机制迥异，但是对于诸如熔断降级这样的功能，在抽象层面上是有迹可循的，不同的框架之间也有思想上的共性，所以 Spring Cloud 为我们提供了标准化的 API，并且对一些主流框架进行自动化适配，以降低开发者的学习成本。

CircuitBreaker

public interface CircuitBreaker {

	default <T> T run(Supplier<T> toRun) {
		return run(toRun, throwable -> {
			throw new NoFallbackAvailableException("No fallback available.", throwable);
		});
	};

	<T> T run(Supplier<T> toRun, Function<Throwable, T> fallback);

}

CircuitBreaker 接口是 Spring Cloud 为我们定义的断路器抽象，它负责去执行一个 Supplier 函数，这里为什么将我们的业务行为抽象为一个 Supplier 呢？因为假设把我们的业务行为当做一个线程，那么它无非是 Runnable 或是 Callable，在函数式编程的层面上，这两种形态是可以用 Supplier 去涵盖的，Runnable 相当于一个 Supplier<Void>，而 Callable<T> 和 Supplier<T> 在函数式表达上是等价的。所以这里 Spring 采用了这样一种抽象的设计，代表对业务行为进行一种包装，嵌入容错方面的逻辑，而 fallback 这个 Function<Throwable, T> 参数代表一种降级策略。

CircuitBreakerFactory

public abstract class CircuitBreakerFactory<CONF, CONFB extends ConfigBuilder<CONF>>
		extends AbstractCircuitBreakerFactory<CONF, CONFB> {

	public abstract CircuitBreaker create(String id);

}

CircuitBreakerFactory 作为 CircuitBreaker 的工厂类，是对于断路器创建过程的抽象。可以看到这里 Spring Cloud 将断路器的配置类完全泛型化，这是因为不同的容错框架在这方面设计迥异， Spring 很难在一个抽象层面上提供一个绝对的标准，不如将其彻底抽象化，为不同的实现方式提供更高的自由度。

Slf4j
RestController
@RequestMapping("user")
public class UserController {

    @Resource
    private UserClient userClient;
    @Resource
    private CircuitBreakerFactory<?, ?> circuitBreakerFactory;

    @GetMapping("getUserInfo")
    public UserInfo getUserInfo(@RequestParam("id") long id) {
        return circuitBreakerFactory.create("getUserInfo").run(() -> userClient.getUserInfo(id), throwable -> {
            log.error("getUserInfo timeout!", throwable);
            return new UserInfo();
        });
    }

    @PostMapping("addUserInfo")
    public long addUserInfo(@RequestBody UserInfo userInfo) {
        return userClient.addUserInfo(userInfo);
    }

}

在使用层面上我们可以看到只需要依赖注入 CircuitBreakerFactory 对象后调用其 create 方法即可包装某一业务方法，使其具有容错效果，同时也可以指定 fallback 逻辑，使用上还是非常灵活的。单纯从调用的代码我们无法判断底层使用的容错框架以及其实现方式，这是 Spring Cloud 的抽象性带来的，使我们脱离具体的底层实现，将注意力集中到业务逻辑中。

Resilience4j 对 Spring Cloud 的适配

Resilience4j 本身是一个轻量化但是功能又颇为全面的容错框架，spring-cloud-starter-circuitbreaker-resilience4j 作为 Spring Cloud 对于 Resilience4j 自动适配包，并没有全面化地使用其全部模块功能，目前只引用了 resilience4j-circuitbreader，resilience4j-timelimiter 两个模块，所以本文主要讨论这两个模块如何在 Spring Cloud 中发挥作用。

Resilience4JCircuitBreaker

public class Resilience4JCircuitBreaker implements CircuitBreaker {
	// ...
	@Override
	public <T> T run(Supplier<T> toRun, Function<Throwable, T> fallback) {
		TimeLimiter timeLimiter = TimeLimiter.of(timeLimiterConfig);
		Supplier<Future<T>> futureSupplier = () -> executorService.submit(toRun::get);
		Callable restrictedCall = TimeLimiter.decorateFutureSupplier(timeLimiter,
				futureSupplier);

		io.github.resilience4j.circuitbreaker.CircuitBreaker defaultCircuitBreaker = registry
				.circuitBreaker(id, circuitBreakerConfig);
		circuitBreakerCustomizer
				.ifPresent(customizer -> customizer.customize(defaultCircuitBreaker));
		Callable<T> callable = io.github.resilience4j.circuitbreaker.CircuitBreaker
				.decorateCallable(defaultCircuitBreaker, restrictedCall);
		return Try.of(callable::call).recover(fallback).get();
	}
	// ...
}

Resilience4JCircuitBreaker 中包含了 Resilience4j 与 Spring Cloud Circuit Breaker 集成的关键性代码，个人认为这是一种对 Decorator Pattern 的灵活运用。首先 Supplier<T> 是我们传入的某一业务行为，如果想嵌入服务熔断降级的相关逻辑，那么必须去包装这个 Supplier<T>，这里主要包含了几个关键过程：

  • 将 Supplier<T> 业务抽象包装成为 Supplier<Future<T>>，将其交给 ExecutorService 线程池执行，当前线程池是允许开发者去自定义的。
  • 通过 TimeLimiter 包装业务行为，嵌入超时熔断逻辑，转换为一个 Callable
  • 从缓存中获取 io.github.resilience4j.circuitbreaker.CircuitBreaker 实例，若缓存中不存在，则进行首次创建。
  • 对 CircuitBreaker 的自定义处理。
  • 使用 CircuitBreaker 对 Callable 进一步包装，嵌入断路器逻辑，产生新的 Callable.
  • 执行最终的 Callable，其中已经嵌入了 Resilience4j 容错相关的逻辑。

TimeLimiter 快速失败

public class TimeLimiterImpl implements TimeLimiter {
	@Override
    public <T, F extends Future<T>> Callable<T> decorateFutureSupplier(Supplier<F> futureSupplier) {
        return () -> {
            Future<T> future = futureSupplier.get();
            try {
                T result = future.get(getTimeLimiterConfig().getTimeoutDuration().toMillis(), TimeUnit.MILLISECONDS);
                onSuccess();
                return result;
            } catch (TimeoutException e) {
                onError(e);
                if (getTimeLimiterConfig().shouldCancelRunningFuture()) {
                    future.cancel(true);
                }
                throw e;
            } catch (ExecutionException e) {
                Throwable t = e.getCause();
                if (t == null) {
                    onError(e);
                    throw e;
                }
                onError(t);
                if (t instanceof Error) {
                    throw (Error) t;
                }
                throw (Exception) t;
            }
        };
    }
}

根据上文我们知道针对一个业务行为，TimeLimiter 会首先对其进行包装，由于处于内层包装，所以 TimeLimiter 的逻辑是最先被执行，可以看到它的代码是比较简单，只是利用 Java 中 Future 对象的等待超时机制判定业务调用是否成功。若等待阈值时间后目前函数依然没有返回，则判定为该次调用失败。这种快速失败的机制在微服务架构中具有重要意义，因为当服务间的 RPC 调用非常普遍时，若系统中的某个服务产生故障，必然导致大量请求阻塞等待在某一链路节点上，此时可能会导致服务端大量线程资源被占用，而线程是宝贵的资源，我们无法忍受线程被某一用户线程长时间占用，所以我们需要设定一个时间阈值，对于超出阈值的调用，快速抛出异常或执行 fallback 备用方案，这是一种取舍，牺牲单一功能的可用性，避免整个系统产生全面性的崩溃，毕竟在一个复杂系统中，总有一些功能在极端情况下是可舍的。

CircuitBreakerStateMachine 状态机

当一个断路器处于闭合状态时，调用链路是畅通的，处于断开状态时，任何调用请求都是被拒绝的，也就是说对于一个断路器而言，它拥有几种状态，在不同的环境下相互切换，同时在某一状态下它的性质是不同的。对于这样的一种应用场景，我们首先想到的设计模式就是 State Pattern 状态机模式，Resilience4j 中的 CircuitBreakerStateMachine 正是采用了状态机的设计思想。

public final class CircuitBreakerStateMachine implements CircuitBreaker {
	// ...
	private final String name;
    private final AtomicReference<CircuitBreakerState> stateReference;
    private final CircuitBreakerConfig circuitBreakerConfig;
    private final CircuitBreakerEventProcessor eventProcessor;
    private final Clock clock;
    private final SchedulerFactory schedulerFactory;
    // ...
}

其中的核心状态就是 stateReference，它是一个 CircuitBreakerState 类型，之所以用 AtomicReference 包装，是为了状态切换时的线程安全。当一个 CircuitBreakerStateMachine 被初始化时，状态机的初始状态为 ClosedState 闭合状态，在此状态下，任何请求都是被放行的。

private interface CircuitBreakerState {

    boolean tryAcquirePermission();

    void acquirePermission();

    void releasePermission();

    void onError(long duration, TimeUnit durationUnit, Throwable throwable);

    void onSuccess(long duration, TimeUnit durationUnit);

    CircuitBreaker.State getState();

    CircuitBreakerMetrics getMetrics();

    /**
     * Should the CircuitBreaker in this state publish events
     * @return a boolean signaling if the events should be published
     */
    default boolean shouldPublishEvents(CircuitBreakerEvent event) {
        return event.getEventType().forcePublish || getState().allowPublish;
    }
    
}

CircuitBreakerState 作为状态抽象了状态机在不同状态下的行为，例如在 ClosedState 下 tryAcquirePermission 是永远成功的，意味着业务调用不会被拦截，如果业务调用成功则触发 onSuccess 回调，若业务调用抛出异常则触发 onError 回调。业务请求的调用结果决定了数据指标的统计，而在 onSuccess 和 onError 中会对数据指标进行分析，判定当前的状态是否需要进行切换。其中主要包含几个决定因素：

  • failureRateThreshold 失败调用率（默认阈值50%）
  • slowCallRateThreshold 慢请求比率（默认阈值100%）
  • waitDurationInOpenState 断路器断开状态维持时间（默认60s）
  • permittedNumberOfCallsInHalfOpenState 断路器在半开状态下允许的请求数量（默认为10）
  • slowCallDurationThreshold 慢请求判断阈值（默认为60s）

这些因素决定了断路器在各个状态下如何切换，以维持系统的容错性：

  • 在 ClosedState 如果失败调用率或慢请求比率被突破，那么状态机会被切换至 OpenState 此时意味着断路器断开，触发熔断。
  • 在 OpenState 状态下，在一定时期内 tryAcquirePermission 是必然失败的，因为此时断路器断开，所有的业务调用都会被拒绝，这个时间范围 retryAfterWaitDuration 是可配置的（默认为60s），这里通过一个 Java8 引入的 Clock 对象记录时间的流逝，当断开状态维持超过所配置的时间范围时，状态机被切换至 HalfOpenState 状态。
  • 在 HalfOpenState 状态下，一定数量的业务调用是被允许的，因为在断路器半开状态，需要取一部分调用样本作为分析依赖，以判定断路器是否应该重新进入闭合状态，在微服务状态下，某些服务有可能是在网络环境不稳定的情况下导致访问失败，或者服务在短暂宕机后又及时恢复了，那么断路器也应该及时地在服务恢复正常状态后放开对该服务的访问。所以对于在 HalfOpenState 下被放行的请求来说，它们也需要被列入数据指标的统计，这一点与 ClosedState 状态下是相同的，若统计结果超出相应的阈值，那么进入 OpenState，反之进入 ClosedState，总而言之 HalfOpenState 属于一种中间状态，它使断路器变得更具弹性。

Fallback 降级策略

在 Spring Cloud Circuit Breaker 中 fallback 策略被定义为 Function<Throwable, T> 类型，它表示一个 Throwable 异常到目前类型 T 的转换函数。这个转换过程由开发者自行定义，有时当一个服务不可用时，我们或许可以从某一个缓存层中获取临时性数据以维持业务的正常运转，或是访问一个备用的服务，又或是记录异常信息，而直接舍弃局部性的功能，这都取决于具体的业务场景，总之 Spring Cloud 为我们提供了足够的灵活性。
