Spring Cache 缓存抽象

Java Spring Cache Caffeine Redis
发布日期:   2020-11-17
文章字数:   2.9k
阅读时长:   12 分

要构建一个高性能高吞吐量的系统，缓存是我们必须考虑的问题。几乎可以这样说，任何一个系统都需要 N 个缓存层来解决数据交互中的 IO 问题，因为磁盘 IO 和内存 IO 在耗时上不是一个数量级的，我们总是更倾向于去读写内存而非磁盘，但内存资源的稀缺性决定了我们无法将所有业务数据加载到缓存中，所以这时候就涉及到了缓存的命中率，淘汰机制等问题，在理想状态下，一个系统的缓存应该以最小的内存代价换取最高的缓存命中率。要达到理想状态需要数据结构及算法方面的支撑，所以我们总会选取一些成熟的方案去实现，例如 Guava Cache，Caffeine，EhCache，Redis 等框架或中间件都是我们会考虑的选择，但是在我们的系统集成这些缓存框架时，其实我们并不希望强依赖于框架本身，毕竟底层的缓存方案在未来存在切换或升级的可能性，例如 Spring Boot 2.0 后将 Guava Cache 升级为 Caffeine，如果我们的业务代码与缓存方案强关联，此时升级成本是高昂的，可谓牵一发而动全身。所以采用一个中间层隔开业务系统与缓存框架的强关联性不失为一个明智的选择。因此 Spring Cache 应运而生，官方文档上称之为 Cache Abstraction 缓存抽象，也就是说 Spring Cache 并不作为一个缓存方案的实现，而是作为一个抽象层，帮助我们嫁接第三方的缓存方案，并且 Spring Cache 采用注解驱动的方式极大程度地简化缓存层面的编码。

Cache Abstraction 缓存抽象

Spring Batch 为了将我们的业务代码与底层缓存实现隔离，必然需要建立一些抽象模型，这些模型需要在缓存层面上具有通用性，同时又应该满足我们大部分的业务需求，虽然 Java 生态内庞杂的缓存方案以及它们个性化的 Native API，决定了 Spring Cache 无法做到彻底的兼容并包，但是 Spring Cache 最大程度上为了我们集成不同的框架提供了便利性。

CacheManager 缓存管理

/**
 * Spring's central cache manager SPI.
 * Allows for retrieving named {@link Cache} regions.
 *
 * @author Costin Leau
 * @since 3.1
 */
public interface CacheManager {

	/**
	 * Return the cache associated with the given name.
	 * @param name the cache identifier (must not be {@code null})
	 * @return the associated cache, or {@code null} if none found
	 */
	Cache getCache(String name);

	/**
	 * Return a collection of the cache names known by this manager.
	 * @return the names of all caches known by the cache manager
	 */
	Collection<String> getCacheNames();

}

首先 Spring 定义了抽象意义上的缓存 Cache，它泛指任何意义上的缓存，针对一个缓存模型，必然有添加，查找，更新，清除等操作。我们的业务系统中往往需要多个这样的缓存模型，所以我们需要一个缓存管理器将它们管理起来，在 Spring Cache 中的这样的模型叫做 CacheManager。CacheManager#getCache 方法会根据缓存名称匹配到对应的缓存，而 CacheManager#getCacheNames 可以让我们获取到该管理器中所有的缓存名称。

CaffeineCacheManager 缓存管理实现

public class CaffeineCacheManager implements CacheManager {

	private final ConcurrentMap<String, Cache> cacheMap = new ConcurrentHashMap<String, Cache>(16);

	private Caffeine<Object, Object> cacheBuilder = Caffeine.newBuilder();

	private CacheLoader<Object, Object> cacheLoader;

	@Override
	public Cache getCache(String name) {
		Cache cache = this.cacheMap.get(name);
		if (cache == null && this.dynamic) {
			synchronized (this.cacheMap) {
				cache = this.cacheMap.get(name);
				if (cache == null) {
					cache = createCaffeineCache(name);
					this.cacheMap.put(name, cache);
				}
			}
		}
		return cache;
	}

	protected Cache createCaffeineCache(String name) {
		return new CaffeineCache(name, createNativeCaffeineCache(name), isAllowNullValues());
	}

	protected com.github.benmanes.caffeine.cache.Cache<Object, Object> createNativeCaffeineCache(String name) {
		if (this.cacheLoader != null) {
			return this.cacheBuilder.build(this.cacheLoader);
		}
		else {
			return this.cacheBuilder.build();
		}
	}
	
}

Caffeine 是一个基于 Java8 开发的提供了近乎最佳命中率的高性能的缓存库。，在 Spring Boot 2.0 之后成为官方推荐的本地缓存方案，我们可以参考下 Spring Cache 适配 Caffeine 的 CaffeineCacheManager，如果我们想要将 Caffeine 作为 Spring Cache 的底层方案，那么我们需要在上下文中装配该 bean，其中的 getCache 展示了创建 CaffeineCache 的过程，CaffeineCache 的初始化参数主要取决于 Caffeine 构建者对象，而这时需要开发者去自定义的。在这里我们可以看到 Spring Cache 倾向于引导开发者使用统一的缓存配置，毕竟 Spring Cache 作为一个抽象层，并不适宜于做太多的个性化设计，如果我们业务实在需要个性化，那么 Spring Cache 建议我们使用底层缓存框架的 Native API，Spring Cache 旨在为我们解决大部分的缓存需求。

注解驱动缓存

Spring Cache 为了降低缓存逻辑对于业务代码的侵入性，使我们仅依赖于一些注解即可实现常规缓存需求，这是一种低耦合的方式，从根本上来说这是 Spring 的 AOP 思想广泛使用于整个家族生态的典型范例。

  • Cacheable 是 Spring Cache 中最常用的注解，用于标注某一方式返回的数据是可被缓存的，而缓存的 key 即为方法参数，该方法被执行一次后，只要缓存尚在有效期内，那么后续针对该方法的调用不会被真正执行，而是在 Spring Cache 的切面中命中缓存返回。

  • CacheEvict 用以标注某一方法执行会导致对应缓存的清除，这往往适用于一些 delete 类型的方法，因为数据库数据被删除后缓存中的数据也应该得到及时清除，保持数据一致性，避免产生不可预知的业务问题。

  • CachePut 的使用场景相对较少，被该注解标注的方法始终会被执行，而执行结果始终被加入到缓存中，这适用于一些 load 类型的方法，有时候我们希望某一方法始终从数据库读取，保证数据的可靠性，同时将最新数据同步到缓存中，方便其他业务场景使用。或者这一类方法可能被我们用于强制刷新缓存。

package com.wanshifu.user.framework.business.cache.common;

import com.wanshifu.user.framework.business.cache.common.key.DemoCacheKey;
import com.wanshifu.user.framework.business.cache.common.value.DemoCacheValue;
import org.springframework.cache.annotation.CacheConfig;
import org.springframework.cache.annotation.CacheEvict;
import org.springframework.cache.annotation.CachePut;
import org.springframework.cache.annotation.Cacheable;

import static com.wanshifu.user.framework.business.cache.configuration.SpringCacheAutoConfiguration.REDIS_CACHE_MANAGER;

/**
 * 缓存示例
 */
@CacheConfig(cacheNames = "demo", cacheManager = REDIS_CACHE_MANAGER)
public class DemoCache {

    /**
     * 缓存中存在，不执行该方法，否则执行并将结果加入缓存
     */
    @Cacheable(key = "#demoCacheKey.id")
    public DemoCacheValue get(DemoCacheKey demoCacheKey) throws InterruptedException {
        Thread.sleep(1000);
        DemoCacheValue demoCacheValue = new DemoCacheValue();
        demoCacheValue.setId(demoCacheKey.getId());
        demoCacheValue.setName(demoCacheKey.getName());
        return demoCacheValue;
    }

    /**
     * 永远执行该方法，并将结果加入缓存
     */
    @CachePut
    public DemoCacheValue load(Long id) throws InterruptedException {
        Thread.sleep(1000);
        DemoCacheValue demoCacheValue = new DemoCacheValue();
        demoCacheValue.setId(id);
        return demoCacheValue;
    }

    /**
     * 清除对应缓存
     */
    @CacheEvict(key = "#demoCacheKey.id")
    public void update(DemoCacheKey demoCacheKey) throws InterruptedException {
        Thread.sleep(1000);
    }

}

值得注意的是 CacheEvict 注解中有一个属性 allEntries，当该属性被置为 true 时，一旦目标方法被调用，Spring Cache 会自动清除该 cacheName 下的所有缓存，这意味着需要同时删除多个 redis key，而通过查看源码发现，底层是使用 KEYS 命令进行模糊匹配，如果此时涉及 KEY 数量较多，并且其中有 big key，由于 redis 是使用单线程模式执行命令，可能造成 redis 主线程长时间阻塞，影响其他客户端请求的执行，所以这个属性需要慎重使用。如果实在需要批量删除 key 的场景，建议直接使用 redis 的 API （例如使用 scan 命令）进行迭代删除。

多种 CacheManager 的并存

在现实的应用场景中我们可能需要多种 Cache 在同个应用中并存，尤其是本地缓存和分布式缓存的并存，对于一些体量较小，实时性要求不高的数据，我们更倾向于将它们缓存到应用本地，毕竟对于本地内存的读写是最为快速的，这个过程不涉及网络 IO 的耗时。其实 Spring Cache 也支持我们达到这样的目的，本人在工作中就做过相应的实践。

通过在 Spirng 容器上下文中装配多个 CacheManager 的 bean，而 Spring Cache 中的注解为我们提供了自定义 cacheManager 的属性，因此我们可以自由地选用当前容器中存在的缓存方案，同时我们的代码只是依赖于 Spring 注解，而不强依赖于底层的缓存框架，这对于我们系统的升级与迁移都具有莫大的意义。

package com.wanshifu.user.framework.business.cache.configuration;

import com.github.benmanes.caffeine.cache.Caffeine;
import com.wanshifu.user.framework.business.cache.common.DemoCache;
import org.springframework.boot.autoconfigure.AutoConfigureAfter;
import org.springframework.boot.autoconfigure.data.redis.RedisAutoConfiguration;
import org.springframework.cache.CacheManager;
import org.springframework.cache.annotation.CachingConfigurerSupport;
import org.springframework.cache.annotation.EnableCaching;
import org.springframework.cache.caffeine.CaffeineCacheManager;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.data.redis.cache.RedisCacheManager;
import org.springframework.data.redis.connection.RedisConnectionFactory;
import org.springframework.data.redis.core.RedisTemplate;
import org.springframework.data.redis.serializer.GenericToStringSerializer;
import org.springframework.data.redis.serializer.StringRedisSerializer;

import javax.annotation.Resource;
import java.time.Duration;
import java.util.HashMap;
import java.util.Map;
import java.util.concurrent.TimeUnit;

import static com.wanshifu.user.framework.business.cache.constant.CachePrefixConstant.COMMON_CACHE_PREFIX;

/**
 * Spring Cache 缓存配置
 * @see DemoCache 缓存示例
 * @author zhangyifan@wshifu.com
 */
@EnableCaching
@Configuration
public class SpringCacheAutoConfiguration extends CachingConfigurerSupport {

    public static final String DEFAULT_CACHE_MANAGER = "defaultCacheManager";
    public static final String REDIS_CACHE_MANAGER = "redisCacheManager";

    @Resource
    private CacheManager defaultCacheManager;

    @Override
    public CacheManager cacheManager() {
        return defaultCacheManager;
    }

    /**
     * 本地（默认）缓存方案
     * @see <a href="https://github.com/ben-manes/caffeine/wiki/Home-zh-CN">Caffeine</a>
     * @see CaffeineCacheManager
     */
    @Configuration
    static class CustomCaffeineCacheConfiguration {

        @Bean(DEFAULT_CACHE_MANAGER)
        public CacheManager defaultCacheManager(Caffeine<Object, Object> defaultCaffeine) {
            CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
            caffeineCacheManager.setCaffeine(defaultCaffeine);
            return caffeineCacheManager;
        }

        @Bean
        public Caffeine<Object, Object> defaultCaffeine() {
            return Caffeine
                    .newBuilder()
                    .expireAfterWrite(1, TimeUnit.DAYS)
                    .maximumSize(1 << 10);
        }

    }

    /**
     * 分布式缓存方案 Redis
     * @see RedisCacheManager
     */
    @AutoConfigureAfter(RedisAutoConfiguration.class)
    @Configuration
    static class CustomRedisCacheConfiguration {

        private final StringRedisSerializer stringRedisSerializer = new StringRedisSerializer();

        private final GenericToStringSerializer<Object> genericToStringSerializer = new GenericToStringSerializer<>(Object.class);

        @Bean(REDIS_CACHE_MANAGER)
        public RedisCacheManager redisCacheManager(RedisTemplate<Object, Object> redisTemplate) {
            RedisCacheManager redisCacheManager = new RedisCacheManager(redisTemplate);
            redisCacheManager.setUsePrefix(true);
            redisCacheManager.setDefaultExpiration(Duration.ofDays(1).getSeconds());
            // 缓存过期时间自定义，默认缓存 1 天
            Map<String, Long> expires = new HashMap<>();
            expires.put("demo", Duration.ofHours(1).getSeconds());
            redisCacheManager.setExpires(expires);
            redisCacheManager.setCachePrefix(cacheName -> stringRedisSerializer.serialize(COMMON_CACHE_PREFIX + cacheName + ":"));
            return redisCacheManager;
        }

        @Bean
        public RedisTemplate<Object, Object> redisTemplate(RedisConnectionFactory redisConnectionFactory) {
            RedisTemplate<Object, Object> template = new RedisTemplate<>();
            template.setConnectionFactory(redisConnectionFactory);
            template.setDefaultSerializer(new GenericFastJsonRedisSerializer());
            template.setKeySerializer(genericToStringSerializer);
            template.setHashKeySerializer(genericToStringSerializer);
            template.setEnableTransactionSupport(true);
            return template;
        }

    }

}

这是本人在工作实践中对于 Spring Cache 自定义配置，本地缓存方案 Caffeine 与分布式缓存 Redis 并存，可以作为参考代码。个人认为 Spring Cache 这个框架最大的意义在于建立缓存的抽象层，最大程度地降低系统业务代码与缓存框架的耦合性，同时简化我们的编码。但是抽象本身是一把双刃剑，松耦合的同时它为我们个性化缓存配置带来了较大的难度，毕竟市场上主流缓存方案的实现各不相同，Spring Cache 在统一层面上不得不做很多的妥协，求同存异，以维持最大程度的统一性。以我个人的使用经验来看，Spring Cache 可以引入我们的系统作为一种兜底的缓存方案，针对特殊化的业务需求，我们依然应该寻求底层缓存框架的 Native API 去解决。
